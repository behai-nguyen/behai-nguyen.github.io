<a id="config-toml"></a>
<span style="font-size:1.4em;font-weight:bold;">⒈</span> <strong>The 
<a href="https://github.com/behai-nguyen/polyglot_pdf/blob/main/pdf_10_bh_cmark/config/config.toml" 
title="The pdf_10_bh_cmark/config/config.toml file" target="_blank">
<code>pdf_10_bh_cmark/config/config.toml</code></a> File
</strong>

● Added the sections <code>[block_spacing.heading]</code>, 
<code>[block_spacing.paragraph]</code>, <code>[block_spacing.image]</code>, 
<code>[block_spacing.thematic]</code>, and <code>[horizontal_break]</code>.

The <code>[horizontal_break]</code> section enables external configuration of the 
horizontal line’s width and colour.

The <code>[block_spacing.*]</code> sections enable external configuration of the 
vertical spacing above and below block elements. In previous iterations, the ad hoc 
“parsers” preserved newlines in the final text, and those newlines naturally became 
blank lines in the PDF. However, 
<a href="https://spec.commonmark.org/" title="CommonMark Spec" target="_blank">CommonMark</a> 
does not retain newlines.  
Hence the introduction of <code>[block_spacing.*]</code> — a more flexible and precise 
approach that provides far greater control than relying on raw newlines.

<a id="config-block-spacing-image-after"></a>
● <code>[block_spacing.image].after</code> renders 
<code>[layout].image_block_spacing</code> obsolete.  
The entire <code>[layout]</code> section has been removed.

title="The pdf_10_bh_cmark/src/main.rs module | measure_block() function" target="_blank">
<code>measure_block()</code></a> — the changes are: first, block‑height 
calculations now include the <code>[block_spacing.*]</code> configuration values as 
<a href="#config-toml">described</a>; second, the function has been extended to handle 
the new block variants.

● For the <code>PreparedBlock::Header</code> and <code>PreparedBlock::Paragraph</code> 
variants, the function reads the corresponding block‑spacing configuration values and 
passes them to the new 
<a href="#text-block-helper"><code>text_block()</code></a> helper function to perform 
the calculations. 🦀 Since both <code>Header</code> and <code>Paragraph</code> may span 
multiple lines, <code>[block_spacing.*].before</code> applies to the first line, and 
<code>[block_spacing.*].after</code> applies to the last line. This logic is easy to 
follow in the 
<a href="https://github.com/behai-nguyen/polyglot_pdf/blob/main/pdf_10_bh_cmark/src/main.rs#L294-L348" 
title="The pdf_10_bh_cmark/src/main.rs module | text_block() function" target="_blank">
<code>text_block()</code></a> function.

● The <code>PreparedBlock::Image</code> variant remains unchanged, since all calculations 
are performed by the <code>pdf_10_bh_cmark/src/image_layout.rs</code> module, as 
<a href="#image-layout-mod">previously discussed</a>.

● The calculation for the new <code>PreparedBlock::Thematic</code> variant is 
self‑explanatory.

<a id="output-positioned-block-func"></a>
⓺ The final function in the pipeline is 
<a href="https://github.com/behai-nguyen/polyglot_pdf/blob/main/pdf_10_bh_cmark/src/main.rs#L439-L496" 
title="The pdf_10_bh_cmark/src/main.rs module | output_positioned_block() function" target="_blank">
<code>output_positioned_block()</code></a> — it follows the same refactoring pattern.

● At this point, both the <code>Header</code> and <code>Paragraph</code> variants 
are effectively identical: they are simply blocks of text. The 
<a href="https://github.com/behai-nguyen/polyglot_pdf/blob/main/pdf_10_bh_cmark/src/main.rs#L448-L462" 
title="The pdf_10_bh_cmark/src/main.rs module | text() closure" target="_blank">
<code>text()</code></a> closure is refactored from the 
<code>PositionedBlock::Text</code> and <code>PreparedBlock::Text</code> logic in the 
<a href="https://github.com/behai-nguyen/polyglot_pdf/blob/main/pdf_09_image_block/src/main.rs#L391-L403" 
title="The pdf_09_image_block/src/main.rs module | output_positioned_block() function" 
target="_blank"><code>output_positioned_block()</code></a> function from the 
<a href="https://behainguyen.wordpress.com/2026/03/25/rust-pdfs-pango-and-cairo-layout-supporting-image-blocks/#measure-block-func" 
title="Rust: PDFs — Pango and Cairo Layout — Supporting Image Blocks" 
target="_blank">last post</a>.  
<code>Header</code> and <code>Paragraph</code> simply call this closure with identical 
parameters to render text using the pre‑computed geometry.

● For the new 
<a href="https://github.com/behai-nguyen/polyglot_pdf/blob/main/pdf_10_bh_cmark/src/main.rs#L478-L493" 
title="The pdf_10_bh_cmark/src/main.rs module | Thematic arm" target="_blank">
<code>Thematic</code></a> arm — calls are made to 
<a href="https://www.cairographics.org/" title="CairoGraphics" target="_blank">
<code>Cairo</code></a> methods to draw the horizontal line, whose properties are defined 
by the <a href="#config-toml">external configuration attributes</a>.

Since the colour of the horizontal line may differ from the rest of the PDF document, 
the current drawing context must be saved, the colour applied, and then the context 
restored for subsequent blocks. This process was first discussed in the 
<a href="https://behainguyen.wordpress.com/2026/03/06/rust-pdfs-cairo-and-pango-an-introduction-to-image-rendering/#ref-documentation" 
title="Rust: PDFs — Cairo and Pango — An Introduction to Image Rendering"
target="_blank">tenth</a> post.

● The existing <code>Image</code> arm remains unchanged.

With the removal of a <a href="#essay-text-file">duplicate paragraph</a> from the input 
file, and the introduction of the <a href="#config-toml"><code>[block_spacing.*]</code></a> 
configuration, the total number of pages is now 35 (thirty‑five), one page fewer than in 
the previous iteration.

<a id="concluding-remarks"></a>
❻ <strong>What’s Next</strong>

That wraps up this article… Aside from the cosmetic <code>Thematic</code> break, 
there is no major feature added compared to the last iteration. Lists and URLs 
are still not available yet. Were the last two ad hoc “parsers” a waste of time?  
I tend not to think so — they were part of the learning curve. This proper Markdown 
parser is solid progress for the project. It is now ready to be extended to support 
lists and URLs, and I am looking forward to that…

Thanks for reading! I hope this post helps others who are looking to deepen their 
understanding of PDF technology. As always—stay curious, stay safe 🦊
